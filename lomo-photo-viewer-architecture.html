Lomo Photo Viewer — Architecture Overview Tauri v2 Desktop App · Immich Frontend · Lomo Backend

Business Context

🎯 Objectives

  • Desktop-native photo viewer for personal photo libraries
  • Reuse Immich's rich web UI without Immich's heavy backend
  • Support both bundled (local) and remote self-hosted servers
  • Simple single-installer distribution (NSIS)

👥 Users & Value

  • Home users with personal photo collections
  • Self-hosters running lomod on a NAS or server
  • No Immich server setup, Docker, or Postgres required
  • HEIC / HEIF support out-of-the-box on Windows

System Architecture

🖥️ UI Layer — Immich SvelteKit SPA

Loaded from submodules/immich/web/, built as static SPA, bundled into web.zip, extracted to %AppData%\com.lomoware.photoviewer\web at runtime.

SvelteKit 5 (runes) adapter-static @immich/ui Tailwind CSS Socket.IO client Tauri JS bindings (core.invoke)
🔌 API Translation Layer — Proxy

proxy/server.ts on port 3001. Translates Immich REST ↔ lomo-backend. Serves the SPA statically.

Express.js Socket.IO server routes/auth.ts routes/timeline.ts routes/assets.ts routes/albums.ts routes/stubs.ts heic-convert sharp (image resize) node-fetch
🦀 Desktop Shell — Tauri Core (Rust)

src-tauri/src/main.rs launches child processes (lomod, proxy), exposes IPC, manages config.

tauri v2 rfd (file dialog) rusqlite (admin DB) serde / serde_json argon2 (password hash) IPC: get_app_settings IPC: save_app_settings IPC: pick_folder IPC: complete_initial_setup
💾 Storage & Backend — lomo-backend + Filesystem

lomod.exe serves on port 8000 (local) or a user-specified remote URL. Assets identified by filename (not UUID). Status bits encode favorites.

lomod.exe (Go) Merkletree timeline buckets /login · /mount · /assets Photos dir (user-selected) config.json web.zip · sharp.zip (extracted)

Features

📅 Timeline

  • Merkletree-based date bucketing
  • Month-detail fallback for remote hash-only trees
  • Favorites (Status & 8 bit)
  • Album grouping

🖼️ Image Handling

  • Thumbnail generation (sharp)
  • HEIC/HEIF decode via heic-convert
  • EXIF orientation respected
  • Streaming full-resolution originals

🔀 Dual-Mode Backend

  • Local: bundled lomod.exe child process
  • Remote: user-entered server URL
  • Settings page with Browse / Save
  • Proxy restarts on mode switch

🔐 Auth & Session

  • Argon2id password hashing
  • Basic-auth login to lomo-backend
  • Cookie-based session (lomo_session)
  • Per-session token + serverUrl

🚦 Server Status

  • Socket.IO server in proxy
  • on_server_version emitted on connect
  • UI shows "Server Online" + version

⚙️ First-Run Setup

  • Photos-folder picker on first launch
  • Admin user bootstrap via rusqlite
  • config.json persisted in AppData
  • Lomod restart on photos_dir change

Non-Functional Requirements

ConcernApproach
Packaging Single NSIS installer. Resources zipped to web.zip & sharp.zip, extracted on first launch.
Local/Remote isolation Proxy routes always use session.serverUrl. No localhost hardcoding. Tauri IPC guarded in UI.
Startup hygiene Stale lomod.exe killed via taskkill /F /IM lomod.exe before spawning new one.
Path handling Windows \\?\ extended-length prefix stripped by clean_path() for Node.js compatibility.
Build reproducibility build-tauri.ps1 orchestrates web → proxy → Tauri with -SkipWeb / -SkipProxy flags.
Cache staleness Extracted web dir must be deleted when web.zip is updated (documented in CLAUDE.md, enforced by convention).

Deployment

📦 Build Artifacts

  • lomo-photo-viewer.exe — main Tauri binary
  • LomoPhotoViewer_1.0.0_x64-setup.exe — NSIS installer (~91 MB)
  • proxy.exe — embedded in resources
  • lomod/lomod.exe — embedded (from lomoagent.msi)
  • web.zip · sharp.zip — extracted at runtime

🚀 Launch Sequence

  • Kill any stale lomod.exe processes
  • Extract web.zip/sharp.zip to AppData if newer
  • Read/create config.json (backend mode + photos_dir)
  • Spawn lomod.exe (local mode only)
  • Spawn proxy.exe on port 3001
  • Tauri WebView navigates to http://localhost:3001
Prerequisite: src-tauri/resources/lomod/lomod.exe must exist before cargo tauri build — extracted from lomoagent.msi.
Iteration gotcha: after editing Svelte sources, all three steps are mandatory — clean build (rm -rf build .svelte-kit), repackage web.zip to both target/debug/ and resources/, and delete the extracted web dir in AppData. Skipping any step leaves the old UI in place.
